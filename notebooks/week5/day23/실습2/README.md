# FastMCP 실습 2 - AI Host 연동 및 도구 자동 선택 관찰

## 실습 목표

실습 1에서 직접 만든 MCP 서버를 실제 LLM(GPT-5.1)에 연동하고, 자연어 요청을 받은 모델이 서버의 도구 목록 중 **스스로 적절한 도구를 선택·호출**하는 과정을 관찰합니다.
실습 1의 개발자 직접 호출과 AI 자율 선택의 차이를 비교합니다.

## 핵심 개념

### 1. 동적 발견 — 도구 목록은 하드코딩되지 않는다

호스트는 서버에 연결한 뒤 런타임에 도구 목록을 조회합니다.
`host.py`에는 greet, get_current_time 같은 도구 이름이 어디에도 적혀 있지 않습니다.
서버에 도구를 추가하면 호스트 코드 수정 없이 모델이 쓸 수 있는 도구가 늘어납니다. (아래 관찰 포인트 6에서 직접 실험합니다)

### 2. 모델이 보는 것 = 도구 명세뿐

FastMCP가 함수 정의에서 자동 생성한 명세(함수 이름, docstring, 타입 힌트)가 `to_llm_tools`(완성 제공)를 거쳐 그대로 모델에게 전달됩니다.

```python
{
    "type": "function",
    "function": {
        "name": tool.name,               # 함수 이름
        "description": tool.description, # docstring → 모델이 도구를 고르는 단서
        "parameters": tool.inputSchema,  # 타입 힌트 → 입력 스키마
    },
}
```

실행하면 이 명세 전체가 JSON으로 출력됩니다. `server.py`의 docstring과 타입 힌트가 각각 어디에 들어갔는지 직접 확인하세요.

> 모델은 함수 본문을 볼 수 없습니다. `server.py`의 docstring은 사람이 아니라 **AI가 읽는 문서**입니다.

### 3. 선택은 모델이, 실행은 호스트가

| 단계 | 주체 | 설명 |
|------|------|------|
| 도구 목록 조회 | 호스트 | [연동 1] 동적 발견 |
| 도구·인수 결정 | **모델** | 요청의 의미와 도구 명세를 대조해 추론 — [연동 2]의 결과 |
| 실제 실행 | **호스트** | [연동 3] 모델의 선택을 서버에 위임 |
| 최종 답변 생성 | 모델 | 도구 실행 결과를 받아 자연어로 정리 |

모델은 도구 호출을 **요청**할 뿐, 실행하지 않습니다. 실행 직전에 호스트가 검토하거나 거부할 수 있으므로, 이 지점이 호스트가 책임지는 **신뢰 경계**가 됩니다.
이번 실습에서는 모델의 선택을 검증 없이 실행합니다 — 허용 목록, 사용자 승인 같은 실행 통제는 섹션 2(도구 실행 보안 및 권한 제한)에서 다룹니다.

## 실행 방법

```bash
# 1) 환경변수 설정: .env.example을 .env로 복사하고 API 키를 입력
cp .env.example .env

# 2) 비교 기준: 개발자 직접 호출 (실습 1 방식)
uv run python client.py

# 3) AI 호스트 실행: 모델의 도구 자동 선택 관찰
uv run python host.py
```

> 서버 기동 배너에 업그레이드 안내(Update available)가 보여도 무시하세요. 이 실습은 fastmcp 버전이 고정되어 있으며, 임의로 업그레이드하면 실습 환경과 어긋날 수 있습니다.

## 관찰 포인트

1. **같은 코드, 다른 요청** — `host.py`에는 "greet을 호출하라"는 코드가 없습니다. 모델이 요청의 의미를 해석해 도구를 골랐습니다.
2. **인수도 모델이 만든다** — "엘리스(Elice)에게 인사해 줘"라는 문장에서 `{"name": "Elice"}`를 추출한 것은 모델의 추론 결과입니다.
3. **도구가 필요 없으면 안 쓴다** — 마지막 요청처럼 자신의 지식으로 답할 수 있으면 모델은 도구를 호출하지 않습니다. 도구 사용 여부도 모델의 판단입니다.
4. **한 요청에 도구 여러 개** — 세 번째 요청에서 모델은 도구 호출 두 개를 한 번에 요청하고, 호스트는 이를 차례로 실행합니다. 단, 이 실습의 호스트는 단일 라운드 구조라서 앞 도구의 결과를 보고 다음 도구를 부르는 연쇄 호출은 지원하지 않습니다.
5. **선택의 근거는 명세** — 실행 시 출력되는 도구 명세 JSON이 모델이 아는 전부입니다. (다음 섹션에서 명세가 선택에 미치는 영향을 직접 실험합니다)
6. **도구 추가 실험 (비교의 핵심)** — `server.py`에 도구 하나를 직접 추가해 보세요. 예를 들어 주사위를 굴리는 `roll_dice`(`random.randint(1, 6)` 반환)를 등록하고, `host.py`는 **한 줄도 고치지 않고** 재실행합니다. 도구 목록과 명세에 새 도구가 자동으로 나타나고, "주사위 굴려 줘" 같은 요청을 추가하면 모델이 골라 씁니다. 같은 일을 `client.py`에서 하려면 호출 코드를 직접 수정해야 합니다 — 이 차이가 동적 발견의 의미입니다.

## 학습 포인트 요약

1. 호스트는 도구 목록 조회 → 명세 전달 → 모델의 선택 감지 → 실행 위임 → 결과 반영 → 최종 응답의 6단계로 동작한다.
2. 도구 목록은 하드코딩이 아니라 런타임에 발견된다 — 서버에 도구를 추가해도 호스트 코드는 그대로다.
3. 모델이 도구에 대해 아는 것은 명세(이름·docstring·타입 힌트)뿐 — 명세가 곧 AI의 인터페이스다.
4. 모델은 도구 호출을 요청할 뿐, 실행의 주체는 호스트다 — 실행 직전이 호스트가 책임지는 신뢰 경계다.
5. 개발자 직접 호출(`client.py`)은 고정된 흐름, AI 자율 선택(`host.py`)은 요청에 따라 달라지는 동적 흐름이다.
