# FastMCP 실습 (섹션 2) - 도구 명세에 따른 AI 선택 변화 관찰

## 실습 목표

같은 동작을 하는 도구 두 개를 **이름과 설명(description)만 다르게** 등록하고,
같은 자연어 요청에 모델이 어느 도구를 고르는지 관찰합니다.
모델은 함수 본문을 보지 못하고 명세(이름·description·입력 스키마)만 보고 고르므로,
'의미론적 명세'가 AI 행동을 얼마나 좌우하는지 직접 체감합니다.

> 섹션 1 실습 2(AI 호스트 연동)에서 쓴 host.py를 그대로 재사용합니다.
> 이번 실습의 변수는 host.py가 아니라 **server.py의 도구 명세**입니다.

---

## 과제

`server.py`의 두 도구는 내부적으로 같은 `_search()`를 호출합니다. 동작은 같습니다.
달라지는 것은 docstring(=모델이 읽는 description)뿐입니다.

| 도구 | 과제 | 의도 |
|------|------|------|
| `process(query)` | **[명세 A]** 모호한 docstring | 무슨 도구인지 추론하기 어렵게 |
| `search_memos(keyword)` | **[명세 B]** 구체적 docstring | 무엇을 어떻게 찾는지 명시 |

`host.py`는 완성본으로 제공됩니다. 명세만 바꿔 가며 `host.py`를 재실행하세요.

---

## 핵심 개념

### 모델이 보는 것 = 명세뿐

`host.py`가 모델에게 보내는 것은 `to_llm_tools`가 만든 명세 JSON입니다.
실행하면 그 JSON이 출력됩니다 — `process`에는 모호한 설명이, `search_memos`에는
구체적인 설명이 들어가 있습니다. 모델은 이 두 설명만 비교해 도구를 고릅니다.

같은 요청 "회의 관련 메모를 찾아줘"에 대해, 잘 작성된 명세(`search_memos`)가 있으면
모델은 그쪽을 더 안정적으로 선택합니다. 명세가 모호하면 선택이 흔들리거나, 엉뚱한 도구를
고르거나, 도구를 아예 쓰지 않기도 합니다.

> 의미론적 명세는 '주석'이 아니라 'AI를 향한 API 문서'입니다.

## 실행 방법

```bash
# 1) 환경변수 설정: .env.example을 .env로 복사하고 API 키를 입력
cp .env.example .env

# 2) AI 호스트 실행: 모델의 도구 선택 관찰
uv run python host.py
```

> 이 실습은 LLM(GPT-5.1) 호출이 필요합니다. `.env`에 과목 전용 API 키가 있어야 합니다.

### 예상 출력 (대표 예시)

```
서버에 등록된 도구: ['process', 'search_memos']

모델에게 전달되는 도구 명세:
[ ... process: "데이터를 처리합니다." / search_memos: "키워드가 제목이나 ..." ... ]

============================================================
사용자 요청    : 회의 관련 메모를 찾아줘.
모델의 선택    : search_memos        <- 구체적 명세 쪽을 선택
모델이 만든 인수 : {'keyword': '회의'}
도구 실행 결과  : ['회의록 2026-06-13']
최종 답변     : '회의'가 들어간 메모로 '회의록 2026-06-13'을 찾았습니다.
```

> 모델의 선택과 답변 문구는 실행마다 달라질 수 있습니다. `process`의 설명을 더 모호하게,
> `search_memos`의 설명을 더 구체적으로 벌릴수록 선택이 한쪽으로 또렷하게 쏠립니다.
> 반대로 두 설명을 비슷하게 두면 선택이 흔들리는 것도 관찰 대상입니다.

---

## 관찰 포인트

1. **명세가 선택을 가른다** — 동작이 같아도 설명이 구체적인 도구를 모델이 선호한다.
2. **모호하면 흔들린다** — 설명이 애매하면 선택이 일관되지 않거나 도구를 안 쓰기도 한다.
3. **이름도 단서다** — `process` vs `search_memos`처럼 이름 자체가 용도를 드러내면 선택에 유리하다.
4. **본문은 무관하다** — 두 도구의 코드는 같다. 선택을 가른 것은 오직 명세다.

---

## 학습 포인트 요약

1. 모델은 도구의 명세(이름·description·스키마)만 보고 고른다 — 함수 본문은 보지 못한다.
2. 구체적이고 의미가 분명한 명세일수록 모델의 선택이 정확하고 일관된다.
3. docstring은 사람을 위한 주석이 아니라 AI를 향한 명세다 — 도구 설계의 핵심 산출물이다.
