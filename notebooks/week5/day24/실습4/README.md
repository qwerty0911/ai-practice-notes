# FastMCP 실습 (섹션 2) - 견고한 예외 대응: 런타임 예외를 AI 피드백으로

## 실습 목표

도구가 **실행되는 동안** 터지는 예외(없는 키, 잘못된 값 등)를 `try/except`로 잡아,
같은 실패를 두 갈래로 분리해 내보냅니다.

- **서버 로그(상세)** — `log.info` / `log.exception` 으로 내부에 상세히 기록(개발자·운영용).
- **모델 메시지(정제)** — raw 예외·스택을 노출하지 말고, "무엇이·왜·어떻게"를 담은 `ToolError`로 변환(AI가 읽고 스스로 고쳐 재시도).

> P5(Pydantic)는 함수 **실행 전** 스키마 검증이고, 이 실습은 함수 **실행 중** 예외 처리입니다. 둘은 상보적입니다.

---

## 먼저 볼 것: 참고 예시 (완성 코드)

`server.py`의 `borrow_book`(도서 대출)은 **완성된 참고 예시**입니다. 메모와 도메인만 다를 뿐,
실행 중 예외(`KeyError`·`ValueError`·예기치 못한 예외)를 `try/except`로 잡아 서버 로그와
정제된 `ToolError`로 분리하는 패턴을 그대로 보여 줍니다. 이 예시를 보고 과제를 구현하세요.

## 과제: `update_memo` 견고하게 만들기

| 위치 | 과제 | 내용 |
|------|------|------|
| `update_memo` | **[런타임 예외 처리]** | 실행 중 예외를 `try/except`로 잡아, 서버 로그(상세) ↔ 모델 `ToolError`(정제)로 분리 |

요구 사항:
- 없는 제목 수정은 실패해야 한다 → 먼저 `_MEMOS[title]`로 조회해 `KeyError`가 나게 하고 잡는다.
- 빈 내용(`new_content.strip()`이 빈 값)도 거부한다.
- 서버 로그엔 상세히(`log.*`), 모델엔 raw 예외 대신 고칠 수 있는 `ToolError`로.
- `borrow_book` 예시 패턴을 따르되 그대로 복사하지 말 것.

## 핵심 개념

### 1. 실행 전 검증 vs 실행 중 예외
Pydantic(P5)은 함수에 들어오기 **전** 스키마를 막는 게이트입니다. 하지만 "없는 메모 수정", "외부 자원
접근 실패"처럼 **실행 중**에야 드러나는 실패가 있습니다. 이건 `try/except`로 잡아야 합니다.

### 2. 로그와 모델 메시지의 분리 (보안·운영)
raw 예외 메시지나 스택 트레이스에는 내부 경로·구조가 담겨 모델(나아가 사용자)에게 노출하면 안 됩니다.
상세는 서버 로그에만 남기고, 모델에는 "무엇이·왜·어떻게"만 담은 `ToolError`를 줍니다.

### 3. 에러는 모델의 다음 행동을 결정하는 단서
잘 쓴 `ToolError`는 모델이 인자를 고쳐 다시 호출하게 만듭니다(자기 교정). 그래서 메시지에
"list_memos로 제목을 확인한 뒤 다시 시도하세요"처럼 **다음 행동**을 담습니다.

## 실행 방법

```bash
uv run python client.py
```

`stdout`에는 모델이 받는 정제된 결과/ToolError가, `stderr`에는 `[서버로그]` 상세 기록이 나옵니다(분리 관찰).

### 예상 출력 (stdout = 모델 수신)

```
== 참고 예시: borrow_book (도서 대출) ==
[정상 대출] 성공: {'status': 'borrowed', 'title': '클린코드'}
[없는 제목] 모델 수신(ToolError): '없는책' 도서가 없습니다. list_books로 제목을 확인한 뒤 다시 시도하세요.
[이미 대출중] 모델 수신(ToolError): '리팩터링'는 이미 대출 중입니다. 반납 후 다시 시도하세요.

== 과제: update_memo (메모 수정) ==
[정상 수정] 성공: {'status': 'updated', 'title': '회의록'}
[없는 제목] 모델 수신(ToolError): '없는메모' 메모가 없습니다. list_memos로 제목을 확인한 뒤 다시 시도하세요.
[빈 내용] 모델 수신(ToolError): 새 내용이 비어 있습니다. 수정할 내용을 입력하세요.

최종 메모: ['회의록']
```

### 서버 로그 (stderr 발췌)

```
[서버로그] INFO update_memo: 없는 제목 title='없는메모'
[서버로그] INFO update_memo: 빈 내용 title='회의록'
```

> 풀이 전(template)에는 `update_memo`가 예외 처리를 안 해, 없는 제목도 새로 만들고 빈 내용도 통과합니다(최종 메모에 '없는메모'가 생김). TODO를 풀면 위처럼 ToolError로 막힙니다. 참고 예시 `borrow_book`은 풀이 전에도 동작합니다.

## 학습 포인트 요약

1. 실행 중에야 드러나는 실패는 `try/except`로 잡는다(실행 전 스키마 검증과 별개).
2. 같은 실패라도 서버 로그(상세)와 모델 메시지(정제 ToolError)는 분리한다 — 내부 정보 노출 방지.
3. `ToolError`에는 "무엇이·왜·어떻게"와 다음 행동을 담아 모델의 자기 교정을 유도한다.
